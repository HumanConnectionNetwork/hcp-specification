# HCP Governance

> **Open standards require open governance.**

The Humanitarian Connection Protocol (HCP) is developed as an open, transparent and community-driven specification.

Our goal is to build an open standard that any humanitarian organization, government, university, volunteer group or software developer can implement without permission.

---

# Guiding Principles

The governance of HCP is based on the following principles:

- Openness
- Transparency
- Neutrality
- Collaboration
- Technical Excellence
- Humanitarian Impact

Every technical decision should prioritize interoperability, simplicity and real-world humanitarian usefulness.

---

# Project Stewardship

The Humanitarian Connection Protocol is currently stewarded by the **Human Connection Network (HCN)** initiative.

At this stage of the project, HCN coordinates:

- maintenance of the specification
- publication of new versions
- review of proposals
- community discussions

As the community grows, governance is expected to evolve into a broader and more representative model.

---

# Open Participation

Anyone may participate by:

- opening Issues
- submitting Pull Requests
- proposing improvements
- reviewing specifications
- improving documentation
- creating reference implementations
- reporting interoperability problems

Participation is open regardless of nationality, organization or professional background.

---

# Decision Making

Whenever possible, decisions should be made through public discussion and community consensus.

When consensus cannot be reached, project maintainers may make temporary decisions in order to keep the project moving forward.

These decisions should always be documented and remain open for future review.

---

# HCP Proposals

Major protocol changes should be introduced through the **Humanitarian Connection Protocol Proposal (HCPP)** process.

Examples include:

- new record types
- breaking changes
- protocol extensions
- synchronization mechanisms
- verification models
- security improvements

Each accepted proposal becomes part of the official specification.

---

# Backward Compatibility

Backward compatibility is considered a core value of HCP.

Breaking changes should be avoided whenever possible.

When unavoidable, migration paths should be clearly documented.

---

# Transparency

All protocol discussions should occur publicly whenever possible.

This includes:

- GitHub Issues
- Pull Requests
- Design discussions
- Proposal reviews

The protocol should never evolve through private decision-making alone.

---

# Vendor Neutrality

HCP belongs to no company, government or commercial platform.

Any organization may implement the protocol.

No implementation is considered "official" simply because it is created by a particular organization.

The specification is the standard.

Implementations are independent.

---

# Community Responsibilities

Community members are encouraged to:

- improve the protocol
- report ambiguities
- review proposals
- create educational content
- build compatible implementations
- help new contributors

Healthy technical discussion is encouraged.

Respectful collaboration is expected.

---

# Future Governance

As adoption grows, HCP governance may evolve toward a more formal structure, including:

- Technical Steering Committee
- Working Groups
- Community Representatives
- Release Managers
- Specification Editors

The governance model will evolve together with the community while preserving the project's core principles.

---

# Long-Term Vision

The Humanitarian Connection Protocol aims to become an open standard for humanitarian information exchange.

Its success will not be measured by the number of official implementations, but by the number of independent systems capable of communicating through a common language.

Open governance is essential to achieving that vision.

---

*"The protocol belongs to everyone who helps build it."*
