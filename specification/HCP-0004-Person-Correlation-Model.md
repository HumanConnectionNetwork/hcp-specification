# 1. Abstract

The Person Correlation Model defines how HCP Nodes correlate multiple Humanitarian Records that may describe the same individual.

HCP does not assign a global identity to any person.

Instead, Nodes analyze structured humanitarian observations and reconstruct probable humanitarian histories by relating independent Humanitarian Records.

The purpose of this model is to transform fragmented humanitarian observations into meaningful humanitarian cases while preserving decentralization and interoperability.

---

# 2. Purpose

Humanitarian emergencies generate multiple observations about the same individual.

A missing person may later be reported by a hospital, a shelter, a rescue team or a family member.

Each observation becomes an independent Humanitarian Record.

The purpose of this specification is to allow Nodes to correlate these observations and present them as a coherent humanitarian case.

---

# 3. Fundamental Principle

A Humanitarian Record represents an observation.

A person is represented by the correlation of multiple observations.

Therefore:

- Humanitarian Records are stored.
- Persons are inferred.

HCP never stores a "Person" as an independent object.

Persons exist only as the result of correlating Humanitarian Records.

---

# 4. Correlation Model

Correlation is a local operation performed by each Node.

Humanitarian Records are never modified during this process.

Instead, the Node analyzes multiple records and estimates whether they probably describe the same individual.

Different Nodes may obtain different correlation results while remaining fully compliant with HCP.

The protocol intentionally allows implementations to improve their correlation methods over time.

---

# 5. Correlation Signals

Correlation should be based exclusively on structured data contained within Humanitarian Records.

Recommended correlation signals include:

- Name similarity
- Approximate age
- Gender
- Physical description
- Last known location
- Event location
- Observation timestamp
- Reported situation
- Last known status
- Family relationships
- Contact references
- Medical references
- Shelter references
- Validator confirmations

Future protocol versions may define additional structured signals.

---

# 6. Excluded Signals

The core HCP specification intentionally excludes binary or media-based information.

Core correlation shall not require:

- Photographs
- Videos
- Fingerprints
- Facial recognition
- Biometric information
- Government identity numbers
- External identity databases

Applications may use additional information internally, but interoperability must never depend on these resources.

---

# 7. Correlation Process

A typical correlation process consists of:

1. Searching candidate Humanitarian Records.
2. Comparing structured fields.
3. Estimating correlation strength.
4. Detecting conflicting information.
5. Ranking possible matches.
6. Identifying the latest observation.
7. Presenting a human-readable humanitarian case.

The protocol does not prescribe any specific algorithm.

---

# 8. Person Candidate

The result of the correlation process is called a Person Candidate.

A Person Candidate is a temporary representation generated by a Node from one or more correlated Humanitarian Records.

A Person Candidate:

- is not a permanent identifier;
- is not globally unique;
- is not stored inside HCP;
- exists only as the result of correlation.

---

# 9. Humanitarian Timeline

When multiple records are successfully correlated, the Node reconstructs a Humanitarian Timeline.

Example:

```text
12 Aug
Missing

↓

13 Aug
Located

↓

13 Aug
Transferred to Hospital

↓

15 Aug
Sheltered

↓

18 Aug
Family Reunified
```

The timeline is generated dynamically.

It is never stored as an independent object.

---

# 10. Last Known Status

Every Person Candidate should expose a Last Known Status.

This corresponds to the most recent Humanitarian Record associated with that Person Candidate.

Typical values include:

- Missing
- Located
- Hospitalized
- Sheltered
- Evacuated
- Reunified
- Deceased

The Last Known Status provides users with the most relevant humanitarian information without requiring them to inspect every individual record.

---

# 11. Search Result Grouping

When searching for a person, Nodes should present results grouped by Person Candidate rather than by individual Humanitarian Record.

The objective is to provide useful humanitarian information rather than exposing users to long lists of disconnected observations.

Each result should represent one probable humanitarian case.

---

# 12. Human-Readable Presentation

A typical search result should contain information such as:

- Display name
- Approximate age
- Last known location
- Last known status
- Date and time of the latest observation
- Confidence level
- Number of correlated Humanitarian Records

Applications remain free to define their own user interface while preserving this semantic meaning.

---

# 13. Confidence

Correlation expresses probability rather than certainty.

Confidence should increase when:

- additional compatible observations are found;
- validator confirmations exist;
- temporal consistency exists;
- geographic consistency exists;
- family relationships are confirmed.

Confidence may decrease when conflicting observations accumulate.

---

# 14. Conflicting Observations

Conflicting information does not invalidate existing Humanitarian Records.

Nodes should preserve every observation while exposing inconsistencies through correlation analysis.

Humanitarian situations frequently evolve under uncertainty.

---

# 15. Local Intelligence

Correlation belongs to the Node.

HCP standardizes Humanitarian Records.

Each Node remains free to improve its correlation algorithms without affecting protocol compatibility.

Future implementations may incorporate statistical models, probabilistic inference or artificial intelligence while remaining HCP compliant.

---

# 16. Privacy

Correlation should use only the minimum humanitarian information necessary.

Its purpose is humanitarian assistance.

It shall never become a global identity system.

---

# 17. Compliance

An implementation complies with this specification when it:

- preserves Humanitarian Records unchanged;
- treats correlation as probabilistic;
- generates Person Candidates dynamically;
- exposes the Last Known Status;
- preserves the complete observation history;
- does not assign global person identifiers.

---

# 18. Summary

The Person Correlation Model enables HCP Nodes to transform independent Humanitarian Records into meaningful humanitarian cases.

Rather than identifying individuals, HCP reconstructs humanitarian histories from decentralized observations.

Users search for people.

HCP stores observations.

The correlation process bridges these two perspectives, allowing humanitarian information to remain decentralized, interoperable and immediately useful during humanitarian emergencies.
