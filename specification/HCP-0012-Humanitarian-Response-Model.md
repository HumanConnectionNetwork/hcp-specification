# 1. Abstract

The Humanitarian Response Model defines how an HCP Node presents humanitarian information after processing a Humanitarian Query.

Rather than exposing raw Humanitarian Records, Nodes generate organized humanitarian responses that summarize correlated observations.

This model improves usability while preserving the decentralized architecture of HCP.

---

# 2. Purpose

The purpose of this specification is to define a common semantic structure for presenting humanitarian information.

Responses should help users understand humanitarian situations without requiring them to interpret multiple independent Humanitarian Records.

---

# 3. Fundamental Principle

Humanitarian Records are exchanged between Nodes.

Humanitarian Responses are presented to Clients.

These are different concepts.

Records are immutable protocol objects.

Responses are dynamic interpretations generated by a Node.

---

# 4. Humanitarian Response

A Humanitarian Response is generated after:

- processing a Humanitarian Query;
- searching Humanitarian Records;
- correlating related observations;
- determining the Last Known Status.

Responses exist only during query execution.

They are never synchronized between Nodes.

---

# 5. Response Structure

A Response may contain one or more Humanitarian Cases.

Each Humanitarian Case represents one probable Subject reconstructed from multiple Humanitarian Records.

---

# 6. Humanitarian Case

A Humanitarian Case should include information such as:

- Display Name
- Approximate Age
- Last Known Status
- Last Known Location
- Date of Latest Observation
- Confidence Level
- Number of Correlated Records

The exact presentation remains implementation-specific.

---

# 7. Multiple Results

A Query may produce multiple Humanitarian Cases.

Nodes should rank results according to local correlation confidence.

Clients should present the highest-confidence cases first.

---

# 8. Empty Results

If no relevant observations are found, the Node should return an empty Humanitarian Response.

This does not imply that the person does not exist.

It indicates only that no matching observations are currently available.

---

# 9. Last Known Status

Every Humanitarian Case should expose the Last Known Status.

This represents the most recent correlated Observation available to the Node.

---

# 10. Humanitarian Timeline

Clients may request the Humanitarian Timeline associated with a Humanitarian Case.

The timeline consists of correlated Humanitarian Records ordered chronologically.

The timeline is generated dynamically.

It is never stored as a permanent object.

---

# 11. Confidence

Every Humanitarian Case should expose a confidence level representing the estimated quality of the correlation.

The protocol does not prescribe any scoring algorithm.

Confidence remains implementation-specific.

---

# 12. Human Readability

Responses should prioritize clarity over technical detail.

End users should receive understandable humanitarian information.

Protocol-specific details remain available for advanced clients when necessary.

---

# 13. Record Access

Advanced Clients may request the underlying Humanitarian Records associated with a Humanitarian Case.

The protocol separates:

- Humanitarian presentation;
- Humanitarian interoperability.

---

# 14. Extensibility

Future specifications may define:

- additional response types;
- summary statistics;
- humanitarian alerts;
- recommendation models;
- AI-assisted responses.

---

# 15. Compliance

A compliant implementation shall:

- generate Humanitarian Responses from Humanitarian Queries;
- preserve Humanitarian Record integrity;
- expose Last Known Status;
- distinguish Humanitarian Records from Humanitarian Responses.

---

# 16. Summary

The Humanitarian Response Model defines how HCP Nodes transform decentralized Humanitarian Records into meaningful humanitarian information.

Rather than exposing raw protocol data, Nodes reconstruct humanitarian cases, determine the Last Known Status and present organized responses that assist humanitarian decision making while preserving interoperability and decentralization.
