# 1. Abstract

The Humanitarian Response Model defines how an HCP Node transforms internal Correlation Candidates into Humanitarian Cases presented to Clients.

Rather than exposing raw Humanitarian Records, Nodes generate organized humanitarian responses that summarize correlated observations in a human-readable form.

This model improves usability while preserving the decentralized architecture of HCP.

---

# 2. Purpose

The purpose of this specification is to define a common semantic model for presenting humanitarian information.

Responses should help users understand humanitarian situations without requiring them to interpret multiple independent Humanitarian Records.

---

# 3. Fundamental Principle

Humanitarian Records are exchanged between Nodes.

Correlation Candidates are generated internally by the Correlation Engine.

Humanitarian Cases are generated by the Response Engine.

Humanitarian Responses are presented to Clients.

These four concepts represent different layers of the protocol.

---

# 4. Humanitarian Response

A Humanitarian Response is generated from one or more Correlation Candidates produced by the Correlation Engine.

The Response Engine transforms these internal objects into Humanitarian Cases suitable for presentation to Clients.

Humanitarian Responses are never exchanged between Nodes.

Only Humanitarian Records are exchanged through HCP.

---

# 5. Response Structure

A Humanitarian Response may contain one or more Humanitarian Cases.

Each Humanitarian Case is generated from exactly one Correlation Candidate.

Multiple Correlation Candidates may exist for the same Humanitarian Query.

The Response Engine selects, orders and formats the most relevant Humanitarian Cases before returning the Humanitarian Response.

---

# 6. Humanitarian Case

A Humanitarian Case is the human-readable representation of one Correlation Candidate.

Its purpose is to summarize humanitarian information in a format suitable for end users.

A Humanitarian Case is not exchanged between Nodes.

It exists only inside Humanitarian Responses.

A typical Humanitarian Case should include:

- Display Name
- Approximate Age
- Last Known Status
- Last Known Location
- Date of Latest Observation
- Confidence Level
- Number of Correlated Humanitarian Records

Applications remain free to define their own visual presentation.

---

# 7. Multiple Results

A Humanitarian Query may generate multiple Correlation Candidates.

Nodes should rank those Correlation Candidates before generating Humanitarian Cases.

Clients should present Humanitarian Cases according to the ranking determined by the Node.

---

# 8. Empty Results

If no relevant observations are found, the Node should return an empty Humanitarian Response.

This does not imply that the person does not exist.

It only indicates that no matching humanitarian observations are currently available.

---

# 9. Last Known Status

Every Humanitarian Case shall expose the Last Known Status obtained from its underlying Correlation Candidate.

The Last Known Status represents the most recent humanitarian observation available to the Node.

---

# 10. Humanitarian Timeline

Clients may request the Humanitarian Timeline associated with a Humanitarian Case.

The Humanitarian Timeline is reconstructed from the Humanitarian Records associated with its Correlation Candidate.

The timeline is generated dynamically.

It is never stored as a permanent object.

---

# 11. Confidence

Confidence values belong to Correlation Candidates.

Humanitarian Cases expose those values without modifying them.

The protocol does not prescribe any specific confidence algorithm.

---

# 12. Human Readability

Responses should prioritize clarity over technical detail.

End users should receive meaningful humanitarian information.

Protocol-specific details remain available for advanced Clients when required.

---

# 13. Record Access

Advanced Clients may request the Humanitarian Records associated with a Humanitarian Case.

The protocol intentionally separates:

- Humanitarian interoperability;
- Humanitarian correlation;
- Humanitarian presentation.

---

# 14. Extensibility

Future HCP specifications may define:

- additional response types;
- humanitarian summaries;
- statistical responses;
- recommendation models;
- AI-assisted responses;
- specialized Humanitarian Cases.

---

# 15. Compliance

A compliant implementation shall:

- generate Humanitarian Responses from Humanitarian Queries;
- preserve Humanitarian Record integrity;
- generate Correlation Candidates internally;
- generate Humanitarian Cases from Correlation Candidates;
- expose the Last Known Status;
- distinguish Humanitarian Records from Humanitarian Responses.

---

# 16. Summary

The Humanitarian Response Model defines how HCP Nodes transform internal Correlation Candidates into Humanitarian Cases suitable for presentation to Clients.

Humanitarian Records remain the only information exchanged between Nodes.

Correlation Candidates remain internal to each Node.

Humanitarian Cases represent the final humanitarian information presented to users, allowing decentralized humanitarian observations to become meaningful, organized and actionable information during humanitarian emergencies.
