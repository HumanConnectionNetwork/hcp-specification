# 1. Abstract

The Event Classification Model defines a common vocabulary for describing humanitarian observations within the Humanitarian Connection Protocol (HCP).

Standardized event classifications improve interoperability between independent HCP Nodes while allowing organizations to extend the model according to their operational needs.

The classification system applies to humanitarian observations rather than to individuals or institutions.

---

# 2. Purpose

Different organizations often describe similar humanitarian situations using different terminology.

For example:

- "Found"
- "Located"
- "Rescued"

may refer to similar humanitarian observations.

This specification establishes a shared classification model to improve interoperability and correlation across independent implementations.

---

# 3. Design Principles

The classification model follows these principles:

- Humanitarian observation oriented
- Simple
- Extensible
- Language independent
- Stable over time
- Backward compatible

The protocol standardizes concepts rather than natural language.

---

# 4. Classification Structure

Each Humanitarian Record should contain an Event Classification.

The classification consists of:

- Category
- Event Type
- Optional Subtype

Example:

Category:
Medical Observation

Type:
Hospital Admission

Subtype:
Emergency Admission

---

# 5. Standard Categories

The following categories are defined by HCP Version 1.

## 5.1 Human Observation

Observations directly related to people affected by a humanitarian event.

Examples:

- Missing Person Observation
- Located Person Observation
- Safe Person Observation
- Injured Person Observation
- Deceased Person Observation
- Family Reunification Observation

---

## 5.2 Medical Observation

Observations generated by healthcare providers.

Examples:

- Hospital Admission
- Medical Evaluation
- Emergency Treatment
- Surgical Intervention
- Intensive Care
- Hospital Discharge

---

## 5.3 Rescue Observation

Observations related to rescue operations.

Examples:

- Rescue Initiated
- Rescue Completed
- Evacuation
- Air Rescue
- Water Rescue
- Extraction

---

## 5.4 Shelter Observation

Observations related to temporary accommodation.

Examples:

- Shelter Registration
- Shelter Transfer
- Shelter Departure
- Temporary Accommodation

---

## 5.5 Humanitarian Aid Observation

Observations describing humanitarian assistance.

Examples:

- Food Distribution
- Water Distribution
- Medical Supply Delivery
- Hygiene Kit Distribution
- Clothing Distribution

---

## 5.6 Infrastructure Observation

Observations describing damage to infrastructure.

Examples:

- Building Collapse
- Road Blocked
- Bridge Damage
- Utility Failure
- Flooded Area
- Fire Damage

---

## 5.7 Environmental Observation

Observations describing environmental conditions affecting humanitarian response.

Examples:

- Flood
- Landslide
- Earthquake Damage
- Wildfire
- Storm Damage
- Extreme Heat

---

## 5.8 Logistics Observation

Observations related to operational logistics.

Examples:

- Access Route Open
- Access Route Closed
- Supply Arrival
- Resource Deployment
- Transportation Available

---

## 5.9 Other Observation

For humanitarian situations not yet covered by the standard taxonomy.

Future HCP versions may standardize additional categories.

---

# 6. Event Codes

Every standardized Event Type should have a stable identifier.

Example:

```
HO-001 Missing Person Observation

HO-002 Located Person Observation

HO-003 Safe Person Observation

MO-001 Hospital Admission

MO-002 Hospital Discharge

RO-001 Rescue Initiated

RO-002 Rescue Completed

SO-001 Shelter Registration

HA-001 Food Distribution

IO-001 Building Collapse
```

Codes are language independent.

Applications may display localized names.

---

# 7. Language Independence

Nodes exchange standardized Event Codes.

User interfaces may display:

English

Spanish

Portuguese

French

Arabic

or any other language.

The exchanged semantic meaning remains identical.

---

# 8. Extensions

Organizations may define additional Event Types.

Example:

```
Category:
Medical Observation

Type:
Hyperbaric Treatment
```

Custom Event Types should not modify existing HCP definitions.

Whenever possible they should use vendor-specific namespaces.

Example:

```
hospital.example.org/HyperbaricTreatment
```

---

# 9. Correlation

Event Classification assists the Correlation Candidate algorithm.

Similar event classifications increase the probability that multiple observations describe the same humanitarian situation.

Classification alone does not establish correlation.

---

# 10. Trust

Trust Score remains independent from Event Classification.

Two identical Event Types may have different Trust Scores depending on:

- reporting organization
- observation consistency
- historical reliability

---

# 11. Lifecycle

The Event Classification describes only the humanitarian observation recorded at a specific moment.

Subsequent observations generate new Humanitarian Records with their own classifications.

Historical observations are never modified.

---

# 12. Versioning

Future HCP versions may introduce:

- additional categories
- new Event Types
- deprecated Event Types

Existing Event Codes should never change semantic meaning.

Backward compatibility is strongly recommended.

---

# 13. Implementation Guidelines

Nodes should:

- recognize standard Event Codes
- preserve unknown custom Event Codes
- avoid rejecting valid records solely because an Event Type is unknown

Unknown Event Types remain valid Humanitarian Records.

---

# 14. Summary

The Event Classification Model establishes a common humanitarian vocabulary for HCP.

Rather than standardizing organizations or people, HCP standardizes humanitarian observations.

This shared taxonomy enables reliable synchronization, correlation, search and interoperability while allowing organizations to extend the protocol according to their operational needs.
